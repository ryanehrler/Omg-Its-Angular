Dig Deeper: Modules

An (Angular) Module, otherwise known as NgModule (en-gee-Module), is the basic organization block of angular. Modules glue your app together and give your app a virtual structure. Another useful way of thinking about modules is a container. For example, if you create a component (or whatever) you will then put the component inside a module. More on this later. Also be sure to not confuse NgModules with Typescript Modules or Javascript Modules. That isn't to say Angular doesn't use js modules, it absolutely does! Js modules just aren't very important for this discussion. Let's take a look below at the CoreModule for Omg, It's Angular! application.

Don't get caught up in all the details for modules right now. We will cover the inner workings of modules later. The key takeaway is Modules give us a way to organize our application, and this is done through the metadata you see above (ie. imports, exports, providers).

Module Summary

  • The term NgModule and Module are the same for the scope of an Angular discussion (aka this guide).
  • Modules organize your application.
  • Modules can be thought of as containers for other Angular blocks.

Dig Deeper: Components

Components, Components, Components. Components are the main building block of your angular application. They are the view. They are the alpha. They are the omega ... wait what?!? Okay seriously, components are one of the building blocks you will be working with the most. They contain your html, scss, and the code behind that drives the behavior of that html. A component can be made up of other components. So components are used to build pages as well as the individual parts that make up a page. Let's take a look at the FramelessImageComponent below!

Angular uses a Component Based Architecture and does not technically fall under the MVC or MV* patterns; although parallels can be drawn. If you have not heard of the MVC pattern then no worries. It's not important to understand since Angular doesn't use it.

To put things into perspective the image you see above is an angular component. So what is this component doing and why is it cool .. I mean powerful?

  1. The component is encapsulating all the parts needed to diplay an image in the center of the screen. Html, Typescript, CSS.
  2. The component is re-usable in any part of our application.
  3. The component Easily modifiable. If we want to update functionality or change the look we can modify the code in one place and our entire application will be updated.

Dig Deeper: Services

Last but not least, services. Services manage your business logic as well as giving you a mechanism to communicate between components. But what about Input/Output variables on components? Data binding (input/output) on components is useful for passing information in when the child component needs all or part of the model from the parent. However other situations arise where data binding will not get the job done. I generally default to services for my communication as this allows me to:

  • Create loosely coupled components.
  • Communicate between adjacent components (one's that aren't parent/child).
  • Persist data between components.
  • Persist data between routes/pages.

When do I use Input/Output variables?

  • For a child component which will use/modify part of the parent data object.
  • Events only used between parent child.
    • For example, a click event fired by a component. Usually the only thing interested in a click event is the parent component. Trying to communicate a click event using a service would be unnecessarily complex.

Unfortunately this is one of those things where there are no hard and fast rules. At first you will be confused and unsure. In time the distinction will make more and more sense. We will touch on this again in the services section with some examples.

I just want to make a quick point on business logic and component logic. What is business logic? Business logic is any logic/code which manipulates data or returns information in a way that does not relate to the UI. An example of business logic would be calculating the total cost of a pizza. An example of UI (component) logic is hide x dropdown when y is selected. It's a good practice to keep these seperate - here are some benefits.

  • Helps you share logic between componets.
  • Helps you maintain Single Responsibility.
    • I don't want to dig too deep into the single responsibility principal since you can find a plethora of information on the topic by googling.
    • I can give a quick example of why it's important though.
      • Imagine you build a component which gives all the options for constructing a pizza.
      • Within this component you have a bunch of different UI rules and branching logic to manage the UI.
      • Also within this component you are calculating the cost of the pizza and within this calculate method you set some boolean values which manipulate the UI.
      • Now your boss asks you, hey can we run these calculations over on y page?
      • Or your boss says to you. Hey we didn't like our last design so we redesigned that page and we need to update the existing page with the latest design.
    • How do services help in the above example?
      • Image that same component above but with a PizzaService class.
      • This PizzaService class has a method on it called Calculate(pizzaToCalculate: Pizza).
      • Instead of your component worrying about the calculation it now just calls pizzaService.Calculate().
      • Now when your boss comes to you with either of the above statements you don't have to worry. You can modify the page without worrying about breaking the calculation portion. Conversely you can update the calculation code without worrying about updating a specific page.
  • More easily test your code; otherwise known as Unit Testing. Operative word here is Unit, your tests should only test one unit of code.
  • More easily test your UI code; otherwise known as UI Testing.
